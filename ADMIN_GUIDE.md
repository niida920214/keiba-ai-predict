# 管理者パネル 運用ガイド

管理者パスワードでログインする担当者向けのマニュアルです。

## 0. ログインと画面

- ログイン画面は一般ユーザーと共通。**管理者パスワード**を入力すると
  「👑 管理者としてログイン中」と表示され、「🎯 レース予測」「🛠 管理者パネル」の2タブが使える
- 一般パスワードでは予測タブしか表示されない（管理者パネルの存在も見えない）
- 一度ログインすると**7日間は自動でログイン状態が維持**される（同じブラウザなら再入力不要）
- パスワードは第三者に教えないこと

## 1. パイプラインの3つの作業

| 作業 | 内容 | 所要時間の目安 |
|---|---|---|
| **① データ更新 (main.py)** | netkeibaから新しいレース結果・馬の過去成績・血統・払戻を**差分取得**し、機械学習用の特徴量データを作り直してクラウドへ保存 | 週1回の差分なら**20〜40分**（数ヶ月空くと1時間超） |
| **② モデル学習 (train_model.py)** | 最新データでLightGBMをOptunaでチューニングし直し、モデルを更新 | **2〜5時間**（Optuna試行回数200の場合。詳細オプションで減らせば短縮可） |
| **③ シミュレーション (simulate.py)** | 更新後のモデルで過去データに賭け戦略を適用し、回収率・グラフを算出 | **15〜30分** |

### ①だけで終わったらどうなる？

最新データがクラウドに保存され、予測は問題なく動き続ける（新しい馬・騎手への
対応データも更新される）。ただし**モデル自体は古いまま**なので、最新のトレンド
（馬場変化・騎手の調子など）は反映されない。それを反映するのが②。

### 推奨の運用サイクル

- **毎週末（日曜の全レース終了後）**: ①だけチェックして起動 → 30〜40分後に完了確認
- **月1回程度（またはG1シーズンの区切り）**: ①②③全部チェックして起動
  → 数時間かかるので**寝る前に起動して翌朝確認**が楽
- ③は②とセットで実行するもの（学習し直したモデルの成績を確認する作業）

## 2. パイプラインの起動

> [!TIP]
> セクション上部の「💡 Recommended run」に、実行記録から判断した
> 「次におすすめの実行」が表示される。「おすすめの内容をチェックにセットする」を
> 押すと①〜③のチェックが自動で設定されるので、基本はこれに従えばよい。

1. 「🛠 管理者パネル」→「🔁 パイプライン実行」で、実行したい作業（①〜③）にチェック
2. 必要なら「詳細オプション」で取得期間・Optuna試行回数を調整
3. **「実行者名」に自分の名前を入力**（必須。実行記録に永続的に残る）
4. 「🚀 パイプラインを起動」を押す

**実行中はボタンが無効化され**「⏳ パイプラインが実行中のため、完了するまで
新しい起動はできません。」と表示される。仮に何かの手違いで二重起動されても、
GitHub側の設定で並行実行はされず順番待ちになるため、データが壊れることはない。

## 3. 進捗・完了の確認（「🔄 実行履歴・進捗を更新」ボタン）

パイプラインは裏側（GitHub Actions）で動くため、**画面は自動更新されない**。
状況を知りたくなったらこのボタンを押す（何度押しても害はない）。

実行中は進捗がステップ単位で表示される:

```
Progress — started 07/17 11:30 (JST) / elapsed 17min
✅ Checkout                          ← 完了
✅ Download data from cloud
🔄 Step 1: Data update (main.py)     ← いま実行中
⏳ Step 2: Train model               ← まだ到達していない
⏳ Upload results to cloud
```

> [!NOTE]
> ⏳は「まだ到達していない」の意味で、必ず実行されるわけではない。
> チェックを入れなかった段階は、順番が来た時点で ⏭️（skipped）として飛ばされる。

履歴テーブルの見方:

| 列 | 意味 |
|---|---|
| Started (JST) | 開始時刻（日本時間） |
| Status | `queued`（順番待ち）/ `in_progress`（実行中）/ `completed`（終了） |
| Result | `✅ success`（成功）/ `❌ failure`（失敗） |
| Elapsed | 所要時間（分） |
| Log | GitHub上の詳細ログへのリンク |

**「Status: completed ＋ Result: ✅ success」になったら完了。**

## 3.5 実行記録（📒 永続・消えない）

「🔄 実行履歴・進捗を更新」の履歴（GitHub由来・約90日で消える）とは別に、
**「📒 実行記録（永続）」**欄には実行1回ごとに1行が永久に残る:

| 列 | 意味 |
|---|---|
| Started (JST) | 実行開始日時 |
| Operator | 起動時に入力された実行者名 |
| ① Update / ② Train / ③ Simulate | 各段階の結果（✅ 成功 / ❌ 失敗 / — 未実行） |
| Result | 全体の成否 |
| Elapsed | 所要時間 |
| **Data through** | **データに含まれる最新の開催日**（この日までのレースが反映済み） |

「Data through」が古いままなら①データ更新が必要、という判断に使える。
過剰実行の防止のため、起動前にこの欄で直近の実行状況を確認すること。

## 4. 完了後にやること

1. 実行が完了すると、実行履歴の下に緑色の案内と
   **「⬇ 完了した成果（最新モデル・データ）を取り込んで反映する」ボタン**が表示されるので押す
   → 新しいデータ・モデルが即座に予測と Recommended run へ反映される
   （押し忘れても、アプリの次回再起動時に自動反映される）
2. ③を実行した場合は「📊 シミュレーション結果」→
   「クラウドから最新の結果を取得して表示」で回収率グラフ・サマリを確認

## 5. 失敗した場合（Result: ❌ failure）

1. 履歴テーブルの **Log** リンクを開き、赤い ❌ のステップをクリックしてエラー内容を確認
2. エラー内容をオーナーに共有
3. 一時的なネットワークエラー等なら、もう一度起動すればよい
   （データ取得は差分設計なので、再実行しても二重取得にはならない）

## 6. その他の機能

- **新しいモデル・データのアップロード**: ローカルPCで学習したモデルファイル
  （`lgbm_model.pickle` 等）を手動で反映したいときに使う。通常運用では不要
- **📄 クラウド上のファイル一覧を表示**: クラウドストレージの保存状況の確認用
